# CONTENT-PROMPT.md — инструкция для ИИ-ассистента, который пишет и публикует статьи в блог quar.io

Этот файл — точка входа для любого ИИ-ассистента (Claude, GPT, Cursor, Codex), которому поручили подготовить новую статью или обновить существующую. Если задача звучит как «напиши статью», «обнови материал», «опубликуй пост» — читай этот файл целиком до начала работы.

Файл [CONTENT-PLAN.md](CONTENT-PLAN.md) — стратегия и календарь. Здесь — операционные правила. Расхождения трактуй так: тема и приоритет берутся из CONTENT-PLAN.md, оформление и публикация — из CONTENT-PROMPT.md.

---

## 1. Где живут статьи

- Все статьи лежат в `content/blog/` как Markdown-файлы.
- Имя файла = `slug` статьи: только латиница, цифры и дефис, длина до 64 символов. Пример: `pochemu-pokupatel-stavit-1-3-zvezdy.md`.
- Картинки-обложки — в `public/blog-covers/` (например, `/blog-covers/1-3-stars.jpg`). Если обложки нет, сработает автогенератор (`/api/blog/og/<slug>.png`).
- Никаких других файлов в `content/blog/` создавать не нужно. Статья = один `.md` файл со всем содержимым и метаданными во frontmatter.

## 2. Frontmatter: обязательные и опциональные поля

В начале файла — YAML-frontmatter между `---`:

```yaml
---
title: "Полное название статьи как оно будет видно читателю"
description: "Краткое описание 140–160 символов. Идёт в <meta description>, OG, превью."
slug: "pochemu-pokupatel-stavit-1-3-zvezdy"
publishedAt: "2026-05-14"
updatedAt: "2026-05-14"
cluster: "returns-rating"
pillar: false
readingTime: 8
coverImage: "/blog-covers/1-3-stars.jpg"
tags:
  - возвраты
  - маркетплейсы
  - рейтинг
relatedSlugs:
  - drugaya-statya-iz-klastera
  - eshe-odna-statya
faq:
  - question: "Можно ли исправить уже поставленную 1 звезду?"
    answer: "На WB и Ozon отзыв убирает модерация только при нарушении правил; во всех остальных случаях работает не удаление, а ответ селлера и работа над причиной."
  - question: "Через сколько дней появляются 1-звёздочные отзывы после доставки?"
    answer: "Пик — 3–7 день после получения, когда покупатель столкнулся с настройкой или первым использованием."
---
```

Правила по полям:

- `title` — H1 статьи, попадает в `<title>` и `og:title`. Без `— quar.io` в заголовке: суффикс добавляется автоматически на странице.
- `description` — 140–160 символов, без многоточий и кликбейта. Это то, что покупатель увидит в выдаче поисковика.
- `slug` — обязан совпадать с именем файла без `.md`. Только латиница, понятная транслитерация русских ключевых слов. Никаких эмодзи, спецсимволов, заглавных букв.
- `publishedAt` — дата первой публикации (ISO `YYYY-MM-DD`). Не меняется при обновлении.
- `updatedAt` — дата последнего значимого обновления. Совпадает с `publishedAt` при первой публикации. Обновляй при любых правках текста (опечатки тоже считаются).
- `cluster` — один из кодов: `returns-rating`, `qr-codes`, `instructions`, `marketplace-law`, `post-purchase`, `niche`. См. CONTENT-PLAN.md.
- `pillar` — `true` для pillar-страниц хаба (3000–5000 слов), иначе `false`.
- `readingTime` — округлённое время чтения в минутах (≈200 слов/мин). Считать руками: количество слов / 200.
- `coverImage` — путь от корня публики (`/blog/covers/...`). Опционально: если не задан, OG-картинку сгенерирует сервер.
- `tags` — 2–5 русскоязычных тегов. Без хэштегов, без `#`. Это не SEO-теги, а навигационные.
- `relatedSlugs` — 3–5 slug-ов связанных статей (из того же или смежного кластера). Сюда **обязательно** входит pillar кластера, если эта статья не pillar.
- `faq` — 4–6 вопросов-ответов. Ответ 1–3 предложения. Это разметится как `schema.org/FAQPage` и может попасть в расширенный сниппет.

Если поле опциональное и его нет — просто не пиши его, не оставляй пустым.

## 3. Язык, тон, требования к тексту

Правила из [CLAUDE.md](CLAUDE.md) (раздел «Контент и тексты») распространяются на статьи полностью. Краткая выжимка:

- Только русский язык. Английский — только для технических терминов без устоявшегося русского аналога.
- Не выноси в текст внутренние коды и идентификаторы. Пиши «Модуль гарантии и регистрации», а не «warranty-registration».
- Никаких слов «временно», «в будущем», «MVP», «скоро», «следующим шагом», «пока работает имитация». Если функция не работает — она не упоминается. Если работает — описывай как факт.
- Без рыбы, без шуток, без разговорной лексики («дёргается карта», «по-человечески», «пуляется письмо»). Тон спокойный, деловой, конкретный. Без снисходительности и без маркетингового шума.
- Сообщения и формулировки должны звучать как продуктовый текст, а не как реплика в чате разработчиков.

Дополнительно для блога:

- Не врать. Цифры либо со ссылкой на источник (исследования, открытые отчёты WB/Ozon, ФАС, ЕАЭС), либо как явные оценки («по нашим наблюдениям», «по данным из кабинета селлеров, с которыми мы работали»).
- Не упоминать конкурентов поимённо в негативном ключе. Сравнение по фичам — нейтрально.
- Никаких выдуманных кейсов. Если кейса нет — пиши обобщённо.
- Не использовать слова «революционный», «инновационный», «уникальный», «лидер рынка» и подобные.

## 4. Структура статьи

Сразу после frontmatter — тело в Markdown. Структура жёсткая:

1. **Lead-абзац (60–80 слов)**. Без подзаголовка. Формулирует проблему и аудиторию. Должен отвечать: «о чём статья» и «кому она полезна».
2. **Оглавление** — только для статей > 1500 слов или с `pillar: true`. Создаётся вручную как ненумерованный список ссылок-якорей на H2. Якоря генерируются автоматически из текста H2.
3. **Тело** — H2/H3. Каждый H2 закрывает один search-intent. Внутри H2 — короткие абзацы (2–4 предложения), маркированные/нумерованные списки, таблицы, выделение жирным ключевых тезисов.
4. **CTA в середине** — после ~40% текста. Не баннер, а короткий блок-врезка: «У quar.io есть [конкретный модуль] для этого — посмотреть, как он выглядит на публичной инструкции». Ссылка ведёт на профильный раздел (см. §6), не на главную.
5. **FAQ** — последний H2 статьи, заголовок «Частые вопросы». Дублирует то, что в frontmatter `faq:`. Это нужно одновременно: видимый блок для читателя и микроразметка для поисковика.
6. **CTA в конце** — короткий блок с явным следующим шагом: попробовать в кабинете, посмотреть демо, прочитать pillar кластера. Один CTA, не два.

Не используй:
- H1 в теле (он генерируется из `title`).
- Эмодзи в заголовках и тексте.
- Восклицательные знаки в заголовках.

## 5. Длина и плотность

| Тип | Слов | H2 | Картинки/таблицы |
|---|---|---|---|
| Pillar (`pillar: true`) | 3000–5000 | 6–10 | 4–8 |
| Обычная статья | 1200–2000 | 4–7 | 2–4 |
| Нишевый шаблон | 1500–2500 | 5–8 | 3–5 |

Если по теме нельзя написать минимум — лучше объединить с соседней, чем растягивать водой. Если получается больше максимума — лучше разделить на две и связать через `relatedSlugs`.

## 6. Перелинковка

Это критически важно для топикал-авторитета. Правила:

- В каждой обычной статье — **3–5 внутренних ссылок** на другие статьи блога. Ссылки делаются естественно внутри текста, а не списком в конце.
- **Обязательно** одна из ссылок ведёт на pillar того же кластера (если эта статья не pillar).
- В pillar-странице — ссылки на **все** статьи кластера в соответствующих разделах, плюс на pillar смежного кластера, если контекст подходит.
- **Одна ссылка** на продуктовый раздел quar.io:
  - Тема про QR/печать → `/pricing` или `/` с якорем на блок про QR
  - Тема про инструкции/редактор → `/` (главная с демо)
  - Тема про возвраты/постпокупочный опыт → `/pricing` (где упомянуты модули обратной связи)
  - Тема про требования и закон → `/legal/terms` уместна только в контексте, иначе на pillar
- **Якорные тексты** — описательные, по теме. Не «здесь», не «по этой ссылке», не «читайте также». Пример хорошо: «как считать юнит-экономику возврата». Пример плохо: «подробнее тут».
- Внешние ссылки — только на первоисточники (тексты регламентов, нормативные акты, исследования). Не на блоги конкурентов и не на агрегаторы.
- После того как новая статья опубликована, открой её `relatedSlugs` и проверь: в этих статьях должна появиться обратная ссылка хотя бы в одной (естественно вписанная). Перелинковка двусторонняя — это критично.

## 7. SEO-разметка (генерируется автоматически, твоя задача — дать данные)

Шаблон страницы статьи сам выставит:

- `<title>` = `${title} — quar.io`
- `<meta name="description">` = `description`
- `<link rel="canonical">` = `${appUrl}/blog/${slug}`
- `og:type=article`, `og:title`, `og:description`, `og:url`, `og:image`, `og:site_name=quar.io`
- `twitter:card=summary_large_image`
- `schema.org/Article` JSON-LD: headline, description, datePublished, dateModified, author, publisher, image, mainEntityOfPage
- `schema.org/FAQPage` JSON-LD из `faq:` блока
- `schema.org/BreadcrumbList` JSON-LD: Главная → Блог → Заголовок статьи
- `<meta name="robots" content="index, follow">` (включается автоматически, потому что `/blog` не в noindex-списке `app.vue`)

Что зависит от тебя:
- Качество `description` — он попадает напрямую в выдачу.
- Полнота `faq:` — это шанс на rich snippet.
- Корректность `publishedAt` / `updatedAt` — поисковики используют это для свежести.
- Картинка с правильным соотношением (1200×630 для OG): либо положи руками в `public/blog-covers/`, либо доверься автогенератору.

## 8. Алгоритм публикации (что сделать пошагово)

1. **Уточни тему по CONTENT-PLAN.md.** Найди статью в календаре или в списке кластеров. Если темы нет — добавь сначала в CONTENT-PLAN.md, потом пиши.
2. **Проверь дубли.** Прочитай `content/blog/`: нет ли уже статьи с похожей темой. Если есть — обновляй её, а не пиши новую.
3. **Сгенерируй slug.** Транслитерация по правилу: «Почему 1–3 звезды и что делать» → `pochemu-1-3-zvezdy-i-chto-delat`. Без артиклей, без предлогов длиннее 3 букв. Только латиница и цифры.
4. **Собери `relatedSlugs`.** Просканируй frontmatter существующих статей в том же `cluster`. Выбери 3–5 наиболее тематически близких. **Обязательно** включи pillar.
5. **Напиши черновик.** Соблюдай §3 (тон) и §4 (структура).
6. **Прогони чек-лист §10.**
7. **Сохрани файл** в `content/blog/<slug>.md`.
8. **Обнови двустороннюю перелинковку.** Открой статьи из `relatedSlugs` и добавь в одну-две из них естественную ссылку на новый материал (в теле, в подходящем месте). Не делай это для всех `relatedSlugs` — достаточно одной-двух, иначе будет спам.
9. **Если статья — pillar:** добавь её slug в `relatedSlugs` всех существующих статей того же кластера.
10. **Проверь, что `pnpm typecheck` и `pnpm build` не падают.** Статьи статичны, но если сломан frontmatter — сборка упадёт на парсинге.
11. **Опубликуй коммитом.** Сообщение коммита: `blog: <slug>` для новых, `blog: update <slug>` для правок.

---

## 9. Обновление старых статей

Поисковая выдача любит свежие даты. Раз в 4–6 месяцев проходи по списку статей кластера и:

- Проверяй цифры, ссылки на нормативку, упоминания фич.
- Дополняй FAQ новыми реальными вопросами из поддержки/чатов.
- Обновляй `updatedAt`. `publishedAt` — не трогай.
- Если изменения косметические (1–2 предложения) — можно не обновлять `updatedAt`, чтобы не вводить поисковик в заблуждение.
- Если статья устарела и не обновляется — добавь в frontmatter `archived: true` (страница перестанет показываться в листинге и sitemap, но останется доступной по прямой ссылке, чтобы не плодить 404).

## 10. Чек-лист перед сохранением

Пройди этот список перед тем, как считать статью готовой:

- [ ] Frontmatter валиден: все обязательные поля заполнены, slug совпадает с именем файла.
- [ ] `description` — 140–160 символов, без многоточий, описывает выгоду для читателя.
- [ ] H1 не дублируется в теле (он берётся из `title`).
- [ ] Lead-абзац 60–80 слов, без подзаголовка.
- [ ] Заголовки H2 закрывают разные search-intent, не повторяют друг друга.
- [ ] Минимальная длина выдержана (см. §5).
- [ ] 3–5 естественных внутренних ссылок, среди них pillar кластера.
- [ ] 1 ссылка на профильный продуктовый раздел.
- [ ] FAQ 4–6 пар, ответы 1–3 предложения, в теле статьи есть видимый блок «Частые вопросы».
- [ ] Нет служебной лексики («MVP», «временно», «скоро», «в разработке»).
- [ ] Нет маркетингового шума («революционный», «уникальный», «лидер»).
- [ ] Цифры либо со ссылкой, либо помечены как оценка.
- [ ] Картинка-обложка существует по пути `coverImage`, либо поле не заполнено.
- [ ] `relatedSlugs` указаны и существуют как файлы в `content/blog/`.
- [ ] Двусторонняя перелинковка добавлена в 1–2 статьи из `relatedSlugs`.
- [ ] `pnpm typecheck` зелёный.

## 11. Чего не делать

- Не создавать страницы статей вручную в `pages/blog/`. Все статьи рендерятся через `pages/blog/[slug].vue` из markdown.
- Не использовать Nuxt Content, MDX, MDC и прочие тяжёлые модули — проект использует минималистичный парсер. Если нужна нестандартная разметка — добавь её через обычный HTML внутри Markdown (Markdown это позволяет).
- Не править `server/utils/blog.ts`, `pages/blog/*.vue`, `server/routes/sitemap.xml.ts`, `server/routes/blog/rss.xml.ts` ради новой статьи. Если хочешь новую механику (например, серию из нескольких частей) — сначала обсуди архитектуру.
- Не публиковать статьи с `publishedAt` в будущем без явной пометки и без согласования: они появятся в sitemap и поисковики попытаются их забрать. Если нужен отложенный выход — оставь файл локально и закоммить в день публикации.
