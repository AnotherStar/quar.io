# server-middleware

## Назначение

Модуль `server-middleware` содержит Nitro server middleware для измерения времени обработки API-запросов в ManualOnline. Он логирует общее время выполнения `/api/*`-запросов, время, затраченное на Prisma-запросы, и вычисляет примерное время выполнения прикладного кода.

## Ключевые возможности

- Автоматическое измерение времени выполнения HTTP-запросов к `/api/*`.
- Разделение общего времени запроса на:
  - суммарное время Prisma-запросов;
  - приблизительное время приложения без учёта БД.
- Логирование количества выполненных Prisma-запросов.
- Детализация самых медленных Prisma-запросов для медленных API-вызовов.
- Минимизация log spam: per-query breakdown выводится только для запросов дольше 500 мс.
- Работает как Nitro middleware и не требует изменений в конкретных API-роутах.

## Архитектура

Модуль представлен одним middleware-файлом:

```text
server/
└── middleware/
    └── timing.ts
```

### `server/middleware/timing.ts`

Файл экспортирует обработчик через `defineEventHandler`, который выполняется на стороне Nitro для входящих серверных запросов.

Основной поток работы:

1. Middleware получает текущий URL из `event.node.req.url`.
2. Если URL не начинается с `/api/`, обработчик сразу завершает работу.
3. Для API-запросов сбрасываются накопленные Prisma timing-данные через `resetPrismaTimings()`.
4. Запоминается стартовое время запроса через `performance.now()`.
5. На событие `finish` HTTP-ответа навешивается обработчик.
6. После завершения ответа:
   - вычисляется общее wall-clock время;
   - считываются Prisma timings через `getPrismaTimings()`;
   - суммируется время всех Prisma-запросов;
   - рассчитывается приблизительное время приложения;
   - выводится summary-лог.
7. Если запрос был медленным и содержал Prisma-запросы, дополнительно выводятся до 5 самых медленных DB-операций.

Middleware не изменяет тело запроса или ответа и не влияет на routing. Его роль — наблюдаемость и диагностика производительности.

## API / Интерфейс

### Nitro middleware

Файл:

```text
server/middleware/timing.ts
```

Экспорт:

```ts
export default defineEventHandler(async (event) => {
  // ...
})
```

Это не route-файл и не объявляет отдельный HTTP-эндпоинт. Middleware применяется Nitro автоматически ко входящим запросам.

### Область действия

Middleware обрабатывает только запросы, URL которых начинается с:

```text
/api/
```

Примеры запросов, которые будут измеряться:

```text
GET /api/manuals
POST /api/products
PATCH /api/instructions/123
```

Примеры запросов, которые будут проигнорированы:

```text
/
 /dashboard
/_nuxt/...
/manuals/some-public-page
```

### Формат логов

Для каждого API-запроса после завершения ответа выводится строка:

```text
[timing] METHOD URL → STATUS  total=Xms  db=Yms (N queries)  app=Zms
```

Пример:

```text
[timing] GET /api/manuals → 200  total=184ms  db=72ms (3 queries)  app=112ms
```

Где:

- `total` — общее время обработки запроса;
- `db` — суммарное время Prisma-запросов;
- `queries` — количество Prisma-запросов;
- `app` — разница между `total` и `db`.

Для медленных запросов дольше 500 мс дополнительно выводятся самые дорогие Prisma-запросы:

```text
[timing]   420ms  prisma.manual.findMany ...
[timing]   130ms  prisma.product.findUnique ...
```

В коде выводится не полный текст `target`, а только первые 80 символов:

```ts
q.target.slice(0, 80)
```

## Бизнес-логика

Модуль содержит техническую бизнес-логику наблюдаемости, связанную с производительностью API.

### Правила обработки

- Измеряются только запросы к `/api/*`.
- Перед началом обработки каждого API-запроса Prisma timing-буфер сбрасывается.
- Время запроса считается от момента входа в middleware до события `finish` HTTP-ответа.
- Суммарное время БД считается как сумма `ms` всех записанных Prisma-запросов:

```ts
const dbTotal = t.queries.reduce((s, q) => s + q.ms, 0)
```

- Время приложения рассчитывается приблизительно:

```ts
app = total - dbTotal
```

Это значение помогает отделять задержки базы данных от остального кода приложения: сериализации, авторизации, бизнес-проверок, сетевых операций и работы Nitro.

### Условие детализации медленных запросов

Детализация отдельных Prisma-запросов включается только если одновременно выполняются условия:

```ts
t.queries.length && total > 500
```

То есть:

- в запросе были обращения к Prisma;
- общее время запроса превысило 500 мс.

В этом случае выводятся максимум 5 самых медленных Prisma-запросов, отсортированных по убыванию времени выполнения:

```ts
t.queries.sort((a, b) => b.ms - a.ms).slice(0, 5)
```

### Ограничения алгоритма

Расчёт `app = total - dbTotal` является приблизительным. Если Prisma-запросы выполняются параллельно, сумма их длительностей может быть больше wall-clock времени, и значение `app` может оказаться заниженным или даже отрицательным. В коде это не обрабатывается отдельно.

## Зависимости

### Внутренние зависимости ManualOnline

Модуль зависит от Prisma timing-утилит:

```ts
import { getPrismaTimings, resetPrismaTimings } from '~~/server/utils/prisma'
```

Ожидаемый контракт этих функций по использованию в коде:

- `resetPrismaTimings()` — очищает накопленные данные о Prisma-запросах для текущего запроса или контекста выполнения.
- `getPrismaTimings()` — возвращает объект с массивом `queries`.

По фактическому использованию каждый элемент `queries` должен иметь поля:

```ts
{
  ms: number
  target: string
}
```

Код самих утилит `server/utils/prisma` в предоставленном фрагменте отсутствует, поэтому детали хранения timing-данных не видны.

### Внешние зависимости

- `node:perf_hooks`
  - используется `performance.now()` для точного измерения времени.
- Nitro / h3 runtime
  - используется `defineEventHandler`;
  - обработчик получает `event`;
  - доступ к Node.js request/response идёт через `event.node.req` и `event.node.res`.
- Node.js HTTP response lifecycle
  - используется событие `finish`:

```ts
event.node.res.on('finish', () => {
  // logging after response is sent
})
```

## Примеры использования

### Пример 1. Диагностика медленного API-запроса

Если API-роут ManualOnline для списка инструкций выполняется медленно:

```text
GET /api/instructions
```

в логах сервера можно увидеть:

```text
[timing] GET /api/instructions → 200  total=812ms  db=645ms (6 queries)  app=167ms
[timing]   390ms  prisma.instruction.findMany ...
[timing]   140ms  prisma.section.findMany ...
[timing]   80ms   prisma.product.findUnique ...
```

Такой лог показывает, что основная задержка связана с базой данных, а не с прикладным кодом API-роута.

### Пример 2. Быстрый API-запрос без детализации

Для быстрого запроса:

```text
POST /api/auth/session
```

middleware выведет только summary:

```text
[timing] POST /api/auth/session → 200  total=93ms  db=21ms (1 queries)  app=72ms
```

Так как общее время меньше 500 мс, список отдельных Prisma-запросов не логируется.

## Замечания

- Middleware применяет измерение только к URL, начинающимся с `/api/`. Публичные страницы инструкций, dashboard-страницы и статические ассеты не попадают в эти логи.
- Логирование выполняется через `console.log`, без структурированного JSON-формата и без интеграции с внешней системой observability.
- Детализация Prisma-запросов ограничена пятью самыми медленными операциями.
- `target` Prisma-запроса обрезается до 80 символов, поэтому в логах может не хватить полного контекста для сложных запросов.
- Значение `app` является приблизительным, так как рассчитывается простым вычитанием суммы Prisma timings из общего времени запроса.
- Корректность изоляции timings между параллельными запросами зависит от реализации `getPrismaTimings()` и `resetPrismaTimings()` в `server/utils/prisma`, код которых в предоставленном фрагменте отсутствует.
- В коде нет явного try/catch: middleware полагается на стандартное поведение Nitro и на то, что logging выполняется после завершения ответа в обработчике `finish`.
