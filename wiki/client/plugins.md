# plugins

## Назначение

Модуль `plugins` содержит Nuxt-плагины приложения ManualOnline. В текущей реализации модуль представлен плагином `auth.ts`, который при запуске приложения гидратирует состояние аутентификации пользователя через `useAuthState().refresh()`.

## Ключевые возможности

- Автоматический запуск логики аутентификации при старте Nuxt-приложения.
- Гидратация auth-состояния как на клиенте, так и при SSR.
- Централизованное обновление текущего состояния пользователя до рендера приложения.
- Использование стандартного механизма Nuxt 3 `defineNuxtPlugin`.
- Интеграция с внутренним composable `useAuthState`.

## Архитектура

Структура модуля:

```text
plugins/
└── auth.ts
```

### `plugins/auth.ts`

```ts
// Hydrate auth state on app boot (client + SSR via universal fetch).
export default defineNuxtPlugin(async () => {
  const { refresh } = useAuthState()
  await refresh()
})
```

Файл расположен в директории `plugins`, поэтому Nuxt 3 автоматически регистрирует его как приложение-level plugin.

Плагин выполняется при инициализации Nuxt-приложения:

1. Nuxt загружает `plugins/auth.ts`.
2. Выполняется `defineNuxtPlugin`.
3. Внутри плагина вызывается composable `useAuthState()`.
4. Из composable извлекается метод `refresh`.
5. Метод `refresh()` вызывается асинхронно и ожидается через `await`.
6. После завершения refresh состояние аутентификации считается актуализированным для дальнейшей работы приложения.

Комментарий в коде указывает, что гидратация выполняется:

- на клиенте;
- при серверном рендеринге;
- через universal fetch внутри `refresh()`.

Сам composable `useAuthState` в предоставленном фрагменте не показан, поэтому детали хранения состояния, формат пользователя, обработка ошибок и API-эндпоинты авторизации в рамках этого модуля не определены.

## API / Интерфейс

### Nuxt plugin

Модуль экспортирует Nuxt-плагин по умолчанию:

```ts
export default defineNuxtPlugin(async () => {
  const { refresh } = useAuthState()
  await refresh()
})
```

Плагин не предоставляет публичный API через `nuxtApp.provide` и не регистрирует inject-поля вида `$auth`.

### Используемый composable

Плагин ожидает наличие composable:

```ts
useAuthState()
```

Минимальный контракт, видимый из кода:

```ts
const { refresh } = useAuthState()
await refresh()
```

То есть `useAuthState()` должен возвращать объект с методом `refresh`, который:

- может быть вызван без аргументов;
- возвращает `Promise` или значение, совместимое с `await`;
- обновляет состояние аутентификации приложения.

### HTTP API

В модуле `plugins` нет Nitro route-файлов, поэтому HTTP-эндпоинты здесь не определены.

### Vue props / events / slots

Модуль не содержит Vue-компонентов, поэтому props, events и slots отсутствуют.

## Бизнес-логика

Основная бизнес-логика модуля — синхронизация текущего auth-состояния при загрузке приложения.

Видимые правила:

- Состояние аутентификации обновляется автоматически при boot приложения.
- Обновление выполняется асинхронно.
- Nuxt ожидает завершения `refresh()` перед продолжением жизненного цикла плагина.
- Плагин универсальный: не имеет суффиксов `.client.ts` или `.server.ts`, поэтому может выполняться и на сервере, и на клиенте.
- Логика не содержит условий, проверок ролей, тарифных ограничений или сценариев публикации.

Из кода не видно:

- какие именно данные пользователя загружаются;
- где хранится auth-состояние;
- какой API-роут используется для refresh;
- как обрабатываются ошибки;
- что происходит при неавторизованном пользователе;
- используется ли cookie/session/token-based авторизация.

## Зависимости

### Внутренние зависимости ManualOnline

- `useAuthState` — composable состояния аутентификации. В данном файле используется метод `refresh`.

### Nuxt / Vue зависимости

- `defineNuxtPlugin` — стандартный API Nuxt 3 для регистрации плагинов.
- Автоимпорты Nuxt — в коде отсутствуют явные импорты, значит `defineNuxtPlugin` и `useAuthState` предполагаются доступными через механизм auto imports.

### Внешние сервисы

Прямых обращений к внешним сервисам в коде нет.

Комментарий указывает на использование universal fetch внутри `refresh()`, но конкретный сервис или эндпоинт в данном модуле не указан.

## Примеры использования

### Автоматическая гидратация auth-состояния

Никаких дополнительных действий в страницах или компонентах для первичной загрузки auth-состояния не требуется. Плагин выполняется автоматически Nuxt-приложением при старте.

```ts
// plugins/auth.ts
export default defineNuxtPlugin(async () => {
  const { refresh } = useAuthState()
  await refresh()
})
```

### Использование auth-состояния после инициализации

Компоненты и страницы могут полагаться на то, что refresh был вызван на этапе запуска приложения. Конкретный API `useAuthState` не показан, поэтому пример ниже отражает только общий сценарий и зависит от фактической реализации composable:

```ts
const auth = useAuthState()

// auth-состояние уже было обновлено плагином при старте приложения
```

## Замечания

- Код модуля минимальный и содержит только один Nuxt-плагин.
- Обработка ошибок при `refresh()` в плагине отсутствует. Если `refresh()` выбросит исключение, это может повлиять на инициализацию приложения.
- Плагин выполняется универсально, без разделения на client/server. Это корректно для SSR-гидратации, но требует, чтобы `useAuthState().refresh()` был безопасен для обеих сред.
- Публичный интерфейс через `nuxtApp.provide` не создаётся.
- Детали `useAuthState`, API авторизации, формат пользователя и механика хранения токенов в предоставленном коде отсутствуют.
