# client-middleware

## Назначение

Модуль `client-middleware` содержит клиентские route middleware Nuxt 3 для управления доступом к страницам quar.io. Он проверяет состояние авторизации пользователя перед переходом на защищённые или гостевые маршруты и выполняет редиректы в зависимости от результата проверки.

## Ключевые возможности

- Проверка авторизации пользователя перед открытием защищённых страниц.
- Автоматическое обновление состояния авторизации через `refresh()`, если пользователь ещё не загружен.
- Редирект неавторизованных пользователей на страницу входа.
- Редирект авторизованных пользователей с гостевых страниц в дашборд.
- Использование стандартного механизма `defineNuxtRouteMiddleware` из Nuxt 3.

## Архитектура

Модуль представлен двумя route middleware-файлами в директории `middleware/`:

```text
middleware/
├── auth.ts
└── guest.ts
```

### `middleware/auth.ts`

Middleware для защищённых маршрутов.

```ts
export default defineNuxtRouteMiddleware(async () => {
  const { user, refresh } = useAuthState()
  if (!user.value) await refresh()
  if (!user.value) {
    return navigateTo('/auth/login')
  }
})
```

Логика работы:

1. Получает состояние авторизации через composable `useAuthState()`.
2. Если `user.value` отсутствует, вызывает `refresh()`.
3. После обновления состояния повторно проверяет наличие пользователя.
4. Если пользователь всё ещё не определён, выполняет переход на `/auth/login`.

Этот middleware предназначен для страниц, доступных только авторизованным пользователям, например для дашборда продавца.

### `middleware/guest.ts`

Middleware для гостевых маршрутов.

```ts
export default defineNuxtRouteMiddleware(async () => {
  const { user, refresh } = useAuthState()
  if (!user.value) await refresh()
  if (user.value) return navigateTo('/dashboard')
})
```

Логика работы:

1. Получает состояние авторизации через `useAuthState()`.
2. Если пользователь ещё не загружен, вызывает `refresh()`.
3. Если после обновления пользователь существует, выполняет редирект на `/dashboard`.

Этот middleware предназначен для страниц, которые должны быть доступны только неавторизованным пользователям, например страницы входа или регистрации.

## API / Интерфейс

В модуле нет Nitro API route-файлов, HTTP-эндпоинтов, Vue-компонентов, props, events или slots.

Основной интерфейс модуля — именованные Nuxt route middleware:

| Middleware | Файл | Назначение |
|---|---|---|
| `auth` | `middleware/auth.ts` | Ограничивает доступ к маршруту только авторизованным пользователям |
| `guest` | `middleware/guest.ts` | Ограничивает доступ к маршруту только неавторизованным пользователям |

### Используемый composable

Оба middleware используют composable:

```ts
const { user, refresh } = useAuthState()
```

На основании текущего кода можно зафиксировать следующий ожидаемый контракт:

| Поле / метод | Назначение |
|---|---|
| `user` | reactive-состояние текущего пользователя, вероятно `Ref` |
| `refresh()` | асинхронная функция обновления состояния авторизации |

Точная реализация `useAuthState()` в предоставленном коде отсутствует, поэтому дополнительные детали о формате пользователя, источнике сессии или механизме refresh не определены.

## Бизнес-логика

### Защищённые маршруты

Для маршрутов с middleware `auth` действует правило:

- если пользователь уже есть в состоянии `user.value`, переход разрешён;
- если пользователя нет, выполняется попытка обновить состояние авторизации через `refresh()`;
- если после обновления пользователь всё ещё отсутствует, пользователь перенаправляется на `/auth/login`.

Иными словами, middleware допускает на страницу только авторизованных пользователей.

### Гостевые маршруты

Для маршрутов с middleware `guest` действует правило:

- если пользователь отсутствует, выполняется попытка обновить состояние через `refresh()`;
- если после обновления пользователь найден, он перенаправляется на `/dashboard`;
- если пользователь не найден, переход на гостевую страницу разрешён.

Это предотвращает доступ уже авторизованных пользователей к страницам входа или другим гостевым сценариям.

### Общий алгоритм проверки

Оба middleware используют одинаковый базовый подход:

```ts
if (!user.value) await refresh()
```

Это означает, что состояние авторизации не считается окончательным, пока не выполнена попытка его обновления. Такой подход полезен при первичной загрузке приложения, когда клиентское состояние ещё не инициализировано.

## Зависимости

### Внутренние зависимости quar.io

- `useAuthState()` — composable состояния авторизации. Используется для получения текущего пользователя и обновления auth-состояния.
- Страницы приложения:
  - `/auth/login` — целевой маршрут для неавторизованных пользователей.
  - `/dashboard` — целевой маршрут для авторизованных пользователей, попавших на гостевые страницы.

### Внешние зависимости

- Nuxt 3:
  - `defineNuxtRouteMiddleware` — объявление route middleware.
  - `navigateTo` — выполнение программного перехода между маршрутами.
- Vue reactivity — косвенно используется через `user.value`, что указывает на reactive/ref-состояние.

## Примеры использования

### Защита страницы дашборда

На странице, доступной только авторизованному продавцу, можно подключить middleware `auth`:

```ts
definePageMeta({
  middleware: 'auth'
})
```

При попытке открыть страницу без активной сессии пользователь будет перенаправлен на страницу входа:

```text
/auth/login
```

### Ограничение доступа к странице входа

Для страницы логина можно использовать middleware `guest`:

```ts
definePageMeta({
  middleware: 'guest'
})
```

Если пользователь уже авторизован, он будет перенаправлен в дашборд:

```text
/dashboard
```

## Замечания

- В предоставленном коде отсутствует реализация `useAuthState()`, поэтому нельзя точно определить, как именно выполняется проверка сессии: через cookies, API-запрос, local state или другой механизм.
- В middleware не обрабатываются ошибки `refresh()`. Если `refresh()` завершится исключением, поведение будет зависеть от глобальной обработки ошибок Nuxt/Vue.
- В коде нет проверки ролей, тарифов, прав доступа к конкретным ресурсам или статусов продавца. Middleware проверяет только факт наличия пользователя.
- Пути редиректов захардкожены:
  - `/auth/login`
  - `/dashboard`
- После редиректа на страницу входа не сохраняется исходный URL назначения. Пользователь после логина не будет автоматически возвращён на изначально запрошенную страницу, если это не реализовано в другом модуле.
